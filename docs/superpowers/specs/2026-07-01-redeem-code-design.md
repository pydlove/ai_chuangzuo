# 兑换码功能设计

> 日期:2026-07-01
> 范围:用户端控制台 Console 兑换码入口与弹框
> 类型:前端原型(无后端)

## 1. 背景与目标

控制台目前已有的「获取奖励」入口只有「🎁 邀请有礼」(主动邀请)。
需要补充一条「受赏」路径:用户拿到兑换码后,能在控制台一键兑换,获得:
- **创作币**(累加到 `coinBalance`)
- **会员天数**(延长 `membershipLevel` 对应的有效期)

本次只做前端原型与伪逻辑,数据保存在 `localStorage`。

## 2. 入口位置

- 位于 `ConsoleLayout.vue` 顶部 `.header-right` 内
- 紧邻 🎁 邀请有礼 按钮的**右侧**
- 同款红色胶囊样式(复用 `.console-invite-btn`),emoji 用 `🎟️`,文案「兑换码」

视觉上与「🎁 邀请有礼」形成「福利入口」组,用户一眼可见。

## 3. 弹框 UI

### 3.1 结构

```
┌─────────────────────────────┐
│         🎟️ 兑换码            │
│   输入兑换码兑换奖励          │
│                             │
│  [____________________]     │   ← 输入框,自动 trim + 转大写
│                             │
│  [      立即兑换       ]    │   ← 主按钮,空输入时 disabled
│                             │
│  ⚠️ 错误提示位(条件渲染)     │
└─────────────────────────────┘
```

### 3.2 成功反馈

成功时弹框顶部展示奖励成功条:

```
┌─────────────────────────────┐
│  ✅ 兑换成功 +100 创作币      │   ← 绿色,2 秒后自动关闭弹框
├─────────────────────────────┤
│         🎟️ 兑换码            │
│   ...                        │
```

### 3.3 组件归属

- 弹框、按钮、状态全部写在 `ConsoleLayout.vue`,与「邀请有礼」同源
- 状态变量:`redeemVisible`、`redeemCode`、`redeemLoading`、`redeemError`、`redeemSuccess`
- CSS 类名以 `redeem-` 为前缀,与 `invite-` / `withdraw-` 风格一致

## 4. 交互流程

```
[点击 🎟️ 兑换码]
  └─> redeemVisible = true,自动聚焦输入框
       ↓
[输入兑换码,自动 trim + 转大写]
  └─> 监听 Enter 键 → 等价于「立即兑换」
       ↓
[客户端校验]
  ├─ 空     → 按钮 disabled
  ├─ 长度错 → redeemError = "兑换码格式不正确"
  └─ 通过   → 进入模拟兑换(loading)
       ↓
[模拟兑换]
  ├─ 命中预置码
  │   ├─ 在已使用集合 → "该兑换码已被使用过"
  │   ├─ 创作币类 → coinBalance += N,显示 "兑换成功 +N 创作币"
  │   └─ 会员类   → membership 更新,显示 "兑换成功 +N 天 XX 会员"
  └─ 未命中 → "兑换码无效或已过期"
       ↓
[成功] 顶部绿色提示条 2 秒后自动关闭弹框,重置输入框与状态
[失败] 弹框保持打开,错误提示保持显示,等用户继续输入
```

## 5. 预置测试码

| 兑换码 | 奖励 | 验证目标 |
|---|---|---|
| `COIN100` | 100 创作币 | 创作币基础路径 |
| `COIN500` | 500 创作币 | 大额创作币 |
| `VIP7DAY` | 7 天专业版会员 | 会员基础路径 |
| `VIP30DAY` | 30 天专业版会员 | 长周期会员 |
| `INVALID` | (永远无效) | 错误路径 |

预置码硬编码在前端常量中,与「模拟数据」同一性质。

## 6. 错误处理

| 场景 | 表现 | 文案 |
|---|---|---|
| 输入为空 | 按钮 disabled | — |
| 长度 < 6 或 > 32 | 输入框红边 | "兑换码格式不正确" |
| 已使用过 | 整体红色提示 | "该兑换码已被使用过" |
| 未命中预置码 | 整体红色提示 | "兑换码无效或已过期" |
| 重复点击 | 按钮 disabled + "兑换中..." | — |

错误与成功状态互斥,共用同一展示位(`.redeem-status`)。

## 7. 数据持久化

### 7.1 新增 localStorage 键

- `aichuangzuo_redeem_codes`:`Array<string>` 已使用码集合,用于防重复
- `aichuangzuo_redeem_history`:`Array<{ code, type, reward, redeemedAt }>` 兑换历史(本次不展示在 UI,留作未来扩展)

### 7.2 复用现有键

- `aichuangzuo_coin_balance`(已存在):创作币类直接累加
- `aichuangzuo_membership`(已存在):扩展为对象 `{ level, expiresAt }`,并兼容旧 string 值

### 7.3 初始化

- 已使用集合初始为空数组
- 预置码不持久化,硬编码

## 9. 非目标(YAGNI)

- 不做后端 API、不做真实码库校验
- 不做兑换码生成/管理后台(属于管理端)
- 不做兑换码历史在 UI 上的展示(本次仅持久化,留接口)
- 不做兑换规则弹窗/教程弹窗
- 不做暗色主题适配复用现有 `body[data-theme="dark"]` 规则即可,本设计不单独列
