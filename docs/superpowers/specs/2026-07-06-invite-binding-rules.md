# 邀请人绑定规则

> **范围**：用户端（`project/user`）邀请返利体系  
> **日期**：2026-07-06  
> **状态**：已确认，待开发

## 1. 核心原则

1. **一对一**：一个用户只能绑定**一个**邀请人。
2. **不可变更**：绑定成功后**永久不可修改、不可解绑、不可重新绑定**。
3. **自愿补绑**：注册时未填写邀请人的用户，可在限定时间内补绑一次。
4. **历史不追溯**：补绑邀请人前已经发生的消费/收益，**不计入**邀请人分成。

## 2. 允许绑定的条件

必须**同时满足**以下条件，绑定接口才受理：

| 条件 | 说明 |
|------|------|
| 未绑定过邀请人 | `inviter_user_id` 为空 |
| 在有效时间窗口内 | 建议注册后 **24h ~ 7 天** 内可补绑；注册时填写不受此限制 |
| 未完成核心转化 | 建议限制：未购买套餐、未首充、未提现等（具体以业务定义为准） |
| 邀请关系合法 | 不能是自己、不能是已有的邀请人/上上游（禁止循环） |
| 邀请人存在且有效 | 邀请人账号状态正常，未被封禁 |

## 3. 禁止的绑定关系

后端校验必须拦截以下场景：

| 场景 | 示例 | 处理 |
|------|------|------|
| 自绑定 | A 填 A 自己的邀请码 | 拒绝，提示“不能绑定自己” |
| 互相绑定 | A 绑 B，B 再绑 A | 拒绝，提示“不能互相绑定” |
| 循环绑定 | A ← B ← C，C 再绑 A | 拒绝，提示“不能形成循环邀请关系” |
| 重复绑定 | 已绑定 A 的用户再次调用绑定接口 | 拒绝，提示“已绑定邀请人，不可修改” |
| 超时补绑 | 注册超过有效时间窗口 | 拒绝，提示“邀请人绑定已过期” |
| 已转化用户补绑 | 已购买套餐/首充后补绑 | 拒绝，提示“已完成转化，不可补绑邀请人” |

## 4. 奖励与分成规则

### 4.1 注册时填写邀请人

- 绑定成功后立即生效。
- 后续所有符合规则的消费/收益按邀请返利策略计算。

### 4.2 注册后补绑邀请人

- 绑定成功后生效。
- **绑定前的历史消费、收益、订单不计入邀请人分成。**
- 仅统计绑定时间之后产生的新消费/收益。

### 4.3 奖励发放节奏

建议采用**后置发放**，降低薅羊毛风险：

- 邀请奖励不在绑定瞬间发放。
- 在被邀请人完成真实行为（如首单支付、套餐生效、提现成功）后，再结算给邀请人。

## 5. 接口行为与错误码

新增/复用错误码建议：

| 错误码 | 含义 | 触发场景 |
|--------|------|----------|
| `INVITE_CODE_INVALID` | 邀请码无效 | 邀请码不存在、已失效 |
| `INVITE_SELF_NOT_ALLOWED` | 不能绑定自己 | 用户填了自己的邀请码 |
| `INVITE_CIRCULAR_NOT_ALLOWED` | 不能形成循环邀请 | 目标用户是当前用户的邀请人/上上游 |
| `INVITE_ALREADY_BOUND` | 已绑定邀请人 | 当前用户已有 `inviter_user_id` |
| `INVITE_BINDING_EXPIRED` | 补绑时间窗口已过 | 注册超过允许补绑的时间 |
| `INVITE_CONVERTED_NOT_ALLOWED` | 已转化用户不可补绑 | 已购买套餐/首充/提现 |

## 6. 数据模型建议

在 `u_user` 表新增字段：

```sql
inviter_user_id BIGINT COMMENT '邀请人用户ID',
inviter_bound_at DATETIME COMMENT '邀请人绑定时间',
```

- `inviter_user_id` 为空表示未绑定。
- 绑定后写入值并记录时间，后续不允许更新。
- 如需支持“注册时填写”和“注册后补绑”两种来源，可用 `inviter_bound_at` 区分，无需额外来源字段。

## 7. 边界场景

1. **用户注销/删除**：邀请关系保留，避免注销后重新注册套利。
2. **邀请人被封禁**：绑定关系不变，但后续奖励可暂停发放；新用户仍不能再绑该邀请人。
3. **同一设备/IP 多账号**：建议风控侧做软限制，不直接由绑定接口硬拦截，但可标记审核。
4. **H5 / 小程序 / App 多端**：邀请码体系统一，绑定规则保持一致。
